# 4B SFT Degeneration Problem — Full Analysis

## Background

We are training Qwen3-4B-Base (a 4B parameter language model) to do math reasoning via supervised fine-tuning (SFT). The model should learn to generate `<think>...</think>` reasoning followed by `\boxed{answer}`.

**Key model property**: Qwen3-4B-Base has `tie_word_embeddings=true` (lm_head shares weights with embed_tokens). Qwen3-8B-Base has `tie_word_embeddings=false`.

## Data

All SFT data is generated by Qwen3-8B (teacher model) on math prompts from OpenThoughts3. Format:
```json
{"messages": [{"role": "user", "content": "math problem"}, {"role": "assistant", "content": "<think>\n...reasoning...\n</think>\n\n\\boxed{answer}"}]}
```

Generation parameters (same for all batches): temperature=0.6, top_p=0.95, top_k=20, max_tokens=16384.

### Datasets:
- **79K filtered**: 79,341 samples. 100% have `</think>` + `\boxed{}`. Mean response length: 38K chars (~12K tokens).
- **152K filtered**: 152,479 samples. 100% have `</think>`, 99.99% have `\boxed{}`. Mean response length: 39K chars. This is 79K + 73K new samples (same generation params, different prompts from same source).
- **179K unfiltered**: 179,341 samples. 85% have `</think>`. This is 79K filtered + 100K raw (unfiltered).

## Training Setup

All experiments use:
- Model: Qwen3-4B-Base
- Framework: ms-swift 4.2.0
- Method: Full fine-tuning (not LoRA)
- lr: 8e-5, cosine schedule, warmup 0.1
- max_length: 16384 (packing)
- gradient_checkpointing, flash_attn, liger_kernel
- bf16

## Degeneration Behavior

When the model fails, it shows this pattern:
1. **First ~10K chars**: Correct, coherent mathematical reasoning
2. **~10K-20K chars**: Reasoning becomes repetitive, circular
3. **>20K chars**: Complete degeneration into garbage (`0 0 0 0`, `\.printStackTrace`, random chars)
4. Never generates `</think>` or `\boxed{}`

On **simple problems** (x²-5x+6=0): Even ckpt-600 degenerates (20K chars, no termination)
On **training data prompts**: All checkpoints work correctly (generate `</think>` + `\boxed{}` within ~10K chars)

## Key Observations

1. **v2 works, everything else doesn't** — despite similar/lower loss
2. **8B works with same unfiltered data** — 8B has `tie_word_embeddings=false`
3. **Loss is not predictive of AIME performance** — 179K 3ep has loss 0.194 (lower than v2's 0.201) but AIME 3.3% vs 50%
4. **Data filtering doesn't help** — 152K (100% filtered) behaves same as 179K (85% filtered)
5. **More epochs don't help** — 3ep same as 2ep (both 3.3%)
6. **Even simple problems degenerate** — not just AIME difficulty

## What's Different About v2?

| Parameter | v2 (works) | 152K 3ep (fails) |
|---|---|---|
| Data size | 79K | 152K |
| Data quality | 100% filtered | 100% filtered |
| Generation params | Same | Same |
| Training lr/schedule | 8e-5 cosine 0.1 warmup | Same |
| Global batch | 256 | 256 |
| Epochs | 3 | 3 |
| Total steps | 759 | 1461 |
| Total samples seen | 237K | 456K |
| DeepSpeed | ZeRO-1 | ZeRO-1 |
| GPU count | 16 (dual node) | 8 (single node) |
| per_device_batch | 8 | 8 |
| grad_accum | 2 | 4 |
| Response length distribution | Mean 38K chars | Mean 39K chars |

The ONLY meaningful differences are:
1. **Data size**: 79K vs 152K (more unique prompts)
2. **Total steps**: 759 vs 1461 (2x more steps due to more data)
3. **GPU topology**: 16 GPU dual-node vs 8 GPU single-node (same global batch)
